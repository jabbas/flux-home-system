# Talos 1.13.2 → 1.13.5 Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. This plan performs **live rolling reboots of all three control-plane nodes** — execute tasks strictly in order, one node at a time, never in parallel.

**Goal:** Upgrade all three Talos nodes (talos1–3) from v1.13.2 to v1.13.5 and align the repo + machine configs with the new version.

**Architecture:** Rolling in-place upgrade via `talosctl upgrade` (drains, installs from factory image, reboots, uncordons — one node at a time, etcd leader last). Repo change is a one-line version bump in `bootstrap/site.yaml` (only consumed at fresh-bootstrap time). Post-upgrade, patch `machine.install.image` on all nodes to clear pre-existing drift (configs still reference v1.12.6).

**Tech Stack:** talosctl v1.13.5 (already installed locally), kubectl, factory.talos.dev image for schematic `8137792da74f62f29f1fa6b6b246ed8258608e342206f4eca3c52acd34fda23c`.

**Verified pre-conditions (2026-07-05):**
- v1.13.5 released; factory ISO for this schematic returns HTTP 200
- All nodes Ready on Talos v1.13.2, Kubernetes v1.35.2; `talosctl health` fully green
- etcd healthy, 3 members, leader = talos2 (`16fc8a7bf76e6b8a`) — hence upgrade order talos1 → talos3 → talos2
- Patch releases only (1.13.3–1.13.5): no breaking changes. Note: 1.13.3+ bumps the *default* Kubernetes to 1.36.x — irrelevant for this upgrade (kubelet stays pinned at 1.35.2), but a fresh `ansible-playbook site.yaml` bootstrap with talosctl 1.13.5 would generate configs defaulting to k8s 1.36.x. Out of scope; noted as follow-up.

**Constants used throughout:**

```
IMAGE=factory.talos.dev/metal-installer/8137792da74f62f29f1fa6b6b246ed8258608e342206f4eca3c52acd34fda23c:v1.13.5
talos1 = 10.1.250.11
talos2 = 10.1.250.12   (etcd leader — upgrade LAST)
talos3 = 10.1.250.13
```

**Rollback:** if a node fails to come back after upgrade: `talosctl rollback -n <ip>` boots the previous Talos partition. Disaster recovery: etcd snapshot taken in Task 2 (`~/talos-etcd-2026-07-05.snapshot`), restore per [Talos disaster recovery docs](https://www.talos.dev/v1.13/advanced/disaster-recovery/).

### Task 3: Upgrade talos1 (10.1.250.11)

- [ ] **Step 1: Run the upgrade**

```bash
talosctl upgrade -n 10.1.250.11 \
  --image factory.talos.dev/metal-installer/8137792da74f62f29f1fa6b6b246ed8258608e342206f4eca3c52acd34fda23c:v1.13.5
```

Expected: progress stream (drain → etcd leave/forfeit → install → reboot → boot). Command blocks until the node is back (`--wait` defaults to true, timeout 30m). Typically 5–15 min. Drain-eviction warnings for pods are normal; drain gives up after 5m and proceeds.

If the command fails or times out: **STOP.** Check `talosctl -n 10.1.250.11 dmesg | tail -50` and consider `talosctl rollback -n 10.1.250.11`.

- [ ] **Step 2: Verify Talos version**

Run: `talosctl -n 10.1.250.11 version | grep Tag`
Expected: two lines, both `v1.13.5` (client and server).

- [ ] **Step 3: Verify node and etcd**

Run: `kubectl get node talos1 -o wide`

Run: `talosctl -n 10.1.250.11,10.1.250.12,10.1.250.13 etcd status`
Expected: 3 members, empty `ERRORS` column.
